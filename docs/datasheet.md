# Datasheet for Tenacious-Bench v0.1 (Interim)

## Telescopic Summary (Pushkarna Layer 1)

Tenacious-Bench is a sales-agent evaluation dataset for **workflow-level B2B outreach reliability** in the Tenacious domain. It targets failure modes not well covered by public benches: weak-signal over-assertion, bench over-commitment, tone-marker violations, override/coordination failures, and condescending gap framing. The dataset is intended for evaluator development and critic/judge training (Path B), not direct prospect automation.

## 1) Motivation (Gebru)

Week 10 showed that fluent outputs can still violate high-cost business constraints (brand safety, capacity honesty, confidence calibration). Existing public benchmarks under-grade these Tenacious-specific behaviors. This dataset exists to make those failures machine-gradable and reproducible.

Primary users:
- Evaluation engineers building mechanical sales-output scorers
- ML engineers training critics/judges for rejection filtering
- Research teams studying policy-safe B2B outreach under weak evidence

## 2) Composition (Gebru)

### Dataset size and partitions
- **Total tasks:** 240 (targeted final v0.1 size; within 200–300 requirement)
- **Train:** 120 (50%)
- **Dev:** 72 (30%)
- **Held-out:** 48 (20%)

### Source-mode composition
| Source mode | Count | Share |
|---|---:|---:|
| Trace-derived | 72 | 30.0% |
| Programmatic sweeps | 72 | 30.0% |
| Multi-LLM synthesis | 60 | 25.0% |
| Hand-authored adversarial | 36 | 15.0% |

### Failure-dimension composition (from audit)
| Failure dimension | Count |
|---|---:|
| Weak-signal confidence calibration | 52 |
| Bench commitment safety | 44 |
| Tone-marker / banned-phrase safety | 50 |
| Multi-system override coordination | 38 |
| Non-condescending gap framing | 56 |
| **Total** | **240** |

### Per-mode examples (what a typical task looks like)
- **Trace-derived:** a real Week 10 enrich/compose output transformed into `task + candidate_output + rubric` with trace ID provenance.
- **Programmatic:** templated cold-email tasks varying role-count confidence, sector, and capacity constraints.
- **Multi-LLM synthesis:** adversarial variants generated by routed model families and judge-filtered for verifiability.
- **Hand-authored adversarial:** crafted edge cases (e.g., dual-control CRM override conflicts) that synthesis misses.

## 3) Collection Process (Gebru)

Data sources:
- Week 10 probe library and failure taxonomy
- Week 10 trace logs (`trace_log.jsonl`, `agent_trace_log.jsonl`)
- Tenacious seed documents (style guide, case studies, pricing, discovery transcripts)

Authoring pipeline:
1. Generate candidates across four source modes.
2. Apply judge-filter scoring (coherence, verifiability, rubric clarity).
3. Deduplicate via pairwise near-duplicate checks.
4. Apply contamination checks before sealing held-out.

Human roles:
- One primary author/labeler for interim pass
- Planned second-pass relabel for agreement checks (24h delayed)

## 4) Preprocessing / Cleaning / Labeling (Gebru)

- Normalize text fields to ASCII-safe JSON
- Remove direct private identifiers; keep domain as "Tenacious workflow" only
- Add structured metadata: source mode, failure dimension, difficulty, partition
- Label rubric dimensions with bounded 1–5 scales and machine-checkable constraints

Inter-rater protocol:
- 30-task re-label audit after 24h, target >=80% agreement per dimension
- If under threshold, revise rubric definitions and relabel

## 5) Uses (Gebru)

### Intended uses
- Mechanical evaluator benchmarking
- Critic/judge model training and calibration
- Regression testing of Tenacious-style outreach constraints

### Out-of-scope uses
- Sending unreviewed live prospect outreach directly from raw benchmark outputs
- Any claim that benchmark pass implies legal/compliance approval
- Cross-domain transfer claims (e.g., healthcare/finance policies) without adaptation

## 6) Distribution (Gebru)

- Planned host: HuggingFace dataset repo (`tenacious-bench-v0.1`)
- Format: JSONL per partition + schema + evaluator scripts
- License target: **CC-BY-4.0**
- Rationale: enables open evaluation reuse while preserving attribution

## 7) Maintenance (Gebru)

Update cadence:

Governance checks per release:
